Pintos

Implements the PintOS OS with priority scheduling, system calls, virtual memory, and a growable file system using indexed and sparse allocation.

Setup

Building the Docker Image

üìÅ Make sure you are inside the setting/ directory before building the image:

cd setting

For amd64 (x86_64) ‚Äì Windows

sudo docker build -t pintos-image .

For ARM64 (Apple Silicon ‚Äì macOS)

docker build --platform=linux/amd64 -t pintos-image .
‚ö†Ô∏è Rosetta must be enabled for x86_64 emulation on Apple Silicon. ![Rosetta Setting](https://github.com/user-attachments/assets/b73e6e6e-b851-4611-82ce-3899333feb6e)


Running the Container

üìÅ Before running the container, move to the project root directory (so that the pintos/ folder is mounted correctly):

cd ..

On amd64 (x86_64) ‚Äì Windows

sudo docker run -it -p 80:80 \
  -v $(pwd)/pintos:/root/pintos \
  --name pintos pintos-image

On ARM64 (Apple Silicon ‚Äì macOS)

docker run --platform=linux/amd64 -it -p 80:80 \
  -v $(pwd)/pintos:/root/pintos \
  --name pintos pintos-image


‚ö†Ô∏è First-Time Setup (Install Bochs in the Container)

After the first run, enter the container and run:

cd /root/pintos/src/misc
env SRCDIR=/root/ PINTOSDIR=/root/pintos/ DSTDIR=/usr/local ./bochs-2.2.6-build.sh
cd ..

You only need to do this once, unless the container is deleted.

Once inside the container, you can exit by typing exit or pressing Ctrl+D. This will automatically stop the container as long as no background processes are keeping it alive.

üîÅ Subsequent Runs

To start the container again after it has been created:

docker start -ai pintos

üìÇ Project 1 ‚Äì Threads (Partial Implementation)

Manual: Pintos Project 1

This project involved modifying the thread scheduler and timer mechanisms in PintOS. We implemented the alarm clock and priority-based scheduling, but priority donation was not included as it was optional.

üìå Related Code

  • devices/timer.c ‚Äì Reimplemented timer_sleep() using sleep queue
  • lib/kernel/list.c ‚Äì Priority comparison for thread queue ordering

üìÇ Project 2 ‚Äì User Programs (Full Implementation)

Manual: Pintos Project 2

This project consisted of two parts:

  • 2-1: Argument passing, system call framework, and basic file I/O
  • 2-2: Full implementation of user-level system calls

The final submission includes both phases, with complete support for all required functionality and test cases.

üìå Related Code

  • userprog/process.c ‚Äì Executable loading, argument stack setup, and child process tracking
  • userprog/syscall.c ‚Äì System call interface: read, write, exec, wait, remove, filesize, seek, tell, etc.
  • lib/user/syscall.c ‚Äì User-space syscall interface
  • lib/string.c, threads/thread.c ‚Äì String helpers, per-thread file descriptor table
  • filesys/file.c, filesys/inode.c ‚Äì Backend logic used by syscall layer for file operations

üìÇ Project 3 ‚Äì Virtual Memory (Partial Implementation)

Manual: Pintos Project 3

This project required implementing virtual memory features such as supplemental page tables, demand paging, stack growth, and swapping. Due to time constraints, only a subset of core VM functionality was implemented, focusing on demand paging and stack growth.

üìå Related Code

  • vm/frame.c, vm/frame.h ‚Äì Frame table and eviction policy
  • vm/page.c, vm/page.h ‚Äì Supplemental page table, lazy loading, and memory tracking
  • vm/swap.c, vm/swap.h ‚Äì Swap disk interface and slot management
  • userprog/exception.c ‚Äì Page fault handler
  • userprog/process.c ‚Äì Stack growth and lazy segment loading

üìå All code in the vm/ directory was written from scratch, as the directory is empty by default.
